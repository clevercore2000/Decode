  === Hardware.java: Central Hardware Manager ===

  // Lines 20-38
  public class Hardware {
      // Subsystems
      public SwerveDrive swerveDrive;

      // Hardware abstraction layers
      public SwerveHardware swerveHardware;
      public OuttakeHardware outtakeHardware;
      public IntakeHardware intakeHardware;

      public Hardware(HardwareMap hardwareMap) {
          // Initialize hardware abstraction layers
          swerveHardware = new SwerveHardware(hardwareMap);
          outtakeHardware = new OuttakeHardware(hardwareMap);
          intakeHardware = new IntakeHardware(hardwareMap);

          // Initialize subsystems with their hardware
          swerveDrive = new SwerveDrive(swerveHardware);
      }
  }

  PURPOSE:
  Central initialization point for all robot hardware - acts as a "single
  source of truth" for hardware access across all OpModes.

  IMPLEMENTATION:
  Two-stage initialization pattern:
  1. Stage 1 (lines 31-33): Create hardware abstraction layers
  (SwerveHardware, OuttakeHardware, IntakeHardware)
  2. Stage 2 (line 36): Pass hardware objects to subsystems (SwerveDrive
  gets SwerveHardware)

  WHY THIS DESIGN:
  - Single initialization: OpModes just create new Hardware(hardwareMap)
  instead of initializing 12+ devices individually
  - Dependency injection: Subsystems receive hardware as constructor
  parameters (testable, maintainable)
  - Separation of concerns: Hardware layer knows about physical devices,
  subsystems know about control logic
  - Prevents duplication: Same hardware object used across autonomous,
  teleop, testing OpModes

  KEY DETAILS:
  - Constructor takes HardwareMap from FTC SDK - this is the bridge to
  physical devices
  - Hardware abstraction happens before subsystems are created (dependency
  flow: HardwareMap вҶ’ SwerveHardware вҶ’ SwerveDrive)
  - Public fields allow OpModes to access both raw hardware (swerveHardware)
   and high-level control (swerveDrive)

  CONNECTIONS:
  - Called by every OpMode in their runOpMode() method
  - SwerveHardware (next section) handles actual device initialization
  - SwerveDrive (covered later) uses SwerveHardware to create 4 SwerveModule
   objects

      /**
       * Normalize angle to [-ПҖ, ПҖ] range

      private double normalizeAngle(double angle) {
          while (angle > Math.PI) {
              angle -= 2 * Math.PI;
          }
          while (angle < -Math.PI) {
              angle += 2 * Math.PI;
          }
          return angle;
      }

  PURPOSE:
  Wrap angles to standard [-ПҖ, ПҖ] range (equivalent to [-180В°, 180В°]).

  IMPLEMENTATION:

  - While angle < -ПҖ: Add 2ПҖ
  - Return wrapped angle

  WHY THIS DESIGN:
  - Angle equivalence: 2ПҖ rad = 360В° = 0В° (circular wrap-around)
  - Standard range: PID controllers expect errors in [-ПҖ, ПҖ]
  - Shortest path: Ensures error calculation finds shortest rotation

    - While loops guarantee correct wrapping regardless of input
  - Performance:
    - Usually 0-1 iterations (angles rarely exceed ВұПҖ)
    - Worst case: 2-3 iterations (if angle extremely large)
    - Negligible overhead
  - Alternative implementation (more efficient but less readable):
  return Math.atan2(Math.sin(angle), Math.cos(angle));

  - Algorithm: kinematics.toSwerveModuleStates(chassisSpeeds)

  Lines 113-116: Speed normalization
  - Check if any module speed exceeds MAX_SPEED_METERS_PER_SECOND
  - If so, scale ALL module speeds proportionally
  - Preserves direction/rotation while respecting speed limit

  WHY THIS DESIGN:
  - Kinematics encapsulation: Don't manually solve equations - library does
  it
  - Automatic coordination: One method call gives all 4 wheel commands
  - Desaturation critical: Prevents commanding impossible speeds

  KEY DETAILS:
  - Inverse kinematics math (simplified):
  For each module at position (x, y):

  tangentialVelocity = omega Г— distance_from_center
  tangentialX = -omega Г— y  (perpendicular to radius)
  tangentialY = omega Г— x

  moduleVx = chassisVx + tangentialX
  moduleVy = chassisVy + tangentialY

  moduleSpeed = sqrt(moduleVxВІ + moduleVyВІ)
  moduleAngle = atan2(moduleVy, moduleVx)
  - Example calculation (pure rotation):
    - Command: vx=0, vy=0, omega=1 rad/s
    - FL module position: (+0.1925, +0.1925)
    - FL tangential velocity: omega Г— 0.272m = 0.272 m/s
    - FL velocity vector: (-0.1925, +0.1925) m/s (tangent to circle)
    - FL angle: atan2(0.1925, -0.1925) = 2.356 rad (135В°)
    - FL speed: sqrt(0.1925ВІ + 0.1925ВІ) = 0.272 m/s вң“
  - normalizeWheelSpeeds() algorithm:
  maxSpeed = max(|module[0].speed|, |module[1].speed|, |module[2].speed|,
  |module[3].speed|)

  if (maxSpeed > MAX_SPEED_METERS_PER_SECOND) {
      scale = MAX_SPEED_METERS_PER_SECOND / maxSpeed
      for each module:
          module.speed *= scale  (angle unchanged!)
  }
  - Why preserve ratios?
    - If you just clamp each wheel individually вҶ’ motion direction changes!
    - Example: Command diagonal (forward + strafe) вҶ’ one wheel hits limit
  before others вҶ’ robot veers off course
    - Proportional scaling: Reduces overall speed but maintains intended
  direction
  - Array ordering CRITICAL:
    - moduleStates[0] = Front-Left
    - moduleStates[1] = Front-Right
    - moduleStates[2] = Back-Left
    - moduleStates[3] = Back-Right
    - Must match kinematics constructor AND setModuleStates() method!

  - moduleStates array passed to setModuleStates() (next section)
  - MAX_SPEED_METERS_PER_SECOND from DriveConstants (3.0 m/s)

  вҳ… Insight в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Why inverse kinematics is "inverse":
  Forward kinematics: Given wheel speeds/angles вҶ’ calculate robot velocity
  (used in odometry). Inverse kinematics: Given desired robot velocity вҶ’
  calculate required wheel speeds/angles (used in drive control). It's
  "inverse" because we're solving the equation backwards: instead of "what
  is the robot doing?", we're asking "what should the wheels do to achieve
  this robot motion?". The math involves solving a system of equations with
  one equation per wheel.

  Data Flow (every control loop iteration):

  Critical Concepts:

  Part A: applyDeadband() - Joystick Noise Filtering

  // Lines 114-119
      private double applyDeadband(double value, double deadband) {
          if (Math.abs(value) < deadband) {
              return 0.0;
          }
          return value;
      }

  PURPOSE:
  Zero small joystick values to eliminate drift.

  IMPLEMENTATION:
  - If absolute value < deadband threshold вҶ’ return 0.0
  - Otherwise вҶ’ return original value unchanged

  WHY THIS DESIGN:
  - Simple threshold: Clean cutoff at deadband value
  - No scaling: Values outside deadband unchanged (maintains proportional
  feel)

  KEY DETAILS:
  - Deadband = 0.05: Values in range [-0.05, +0.05] вҶ’ 0.0
  - Effect on control:
    - Joystick at rest: Reads Вұ0.02 (noise) вҶ’ 0.0 (no motion) вң“
    - Joystick at 0.1: Passes through unchanged вҶ’ 0.1 (10% speed)
  - Alternative implementations:
    - Scaled deadband: (value - deadband) / (1 - deadband)
        - Removes deadband AND scales remaining range to [0, 1]
      - Pro: Smooth transition at deadband edge
      - Con: More complex, slightly unnatural feel
    - Current (simple threshold):
        - Pro: Easy to understand, maintains proportional feel
      - Con: Slight "jump" from 0 to 0.05 (usually unnoticeable)
  - Tuning deadband:
    - Too small (0.01): Robot drifts, drivers frustrated
    - Too large (0.15): Controls feel sluggish, hard to make small
  adjustments
    - 0.05 (5%): Sweet spot for most FTC gamepads

  CONNECTIONS:
  - Called for all 3 inputs: forward, strafe, rotation (lines 56-58)
  - JOYSTICK_DEADBAND from ControlConstants (0.05)
  - Critical for preventing unintended motion

  Control Flow (every loop iteration, ~50-100 Hz):
  1. Read gamepad inputs (left stick, right stick)

     вҶ“
  9. Update game mechanisms (outtake/intake)
